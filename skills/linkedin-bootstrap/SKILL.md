## Contract

This skill guarantees:

- **Operator identity** (Phase 1, T2): Profile.csv + Positions.csv +
  Education.csv + Skills.csv + Languages.csv + Email Addresses.csv +
  PhoneNumbers.csv + Registration.csv + Verifications/Verifications.csv +
  Profile Summary.csv are read and the operator self-page at
  `~/brain/people/emanuel.md` is augmented (NOT overwritten —
  user-authored body content above the line is preserved). Each
  distinct past employer becomes (or updates) a stub at
  `~/brain/institutions/<org-slug>.md`; each distinct school becomes
  (or updates) a stub. Operator self-page resolution: read
  `~/.hermes/USER.md operator.brain_slug` field if present; fall back
  to `emanuel.md` heuristic; fail-fast on ambiguity.
- **Network graph** (Phase 2, T2): Connections.csv yields one
  `~/brain/people/<sub-dir>/<slug>.md` stub per LinkedIn connection
  (sub-dir routed per existing apple-contacts-bootstrap conventions
  via known-academic-orgs.yaml + family-group-terms.yaml). D49 alias
  dedup against the existing 5557+ people-pages corpus: a Connection
  row's `linkedin_url` (canonical), then `email`, then DIN 5007-2
  romanized name is checked against existing pages. Match → page is
  updated with LinkedIn URL added to `aliases:`. No match → new stub
  created. LinkedIn-Member placeholder rows skipped silently.
- **Imported-contacts alias-merge** (Phase 3, T2): ImportedContacts.csv
  is processed for ALIAS-MERGE ONLY against existing pages. NEVER
  creates new pages. Rows that don't dedup-match an existing page are
  logged + skipped. (Operator scope decision S191-Q2: ImportedContacts
  is a historical snapshot of phones uploaded to LinkedIn; Apple
  Contacts is the live SSOT for that surface; we want LinkedIn-side
  aliases only.)
- **Network signals** (Phase 4, T2): Endorsement_Given_Info.csv +
  Endorsement_Received_Info.csv + Invitations.csv + Company Follows.csv +
  Member_Follows.csv produce relationship facts + timeline entries on
  existing/new people + institutions pages. May create new stubs for
  entities not seen in Phase 2 (e.g., a company you follow but have no
  direct connection to).
- **Activity timeline** (Phase 5, T2): Comments_*.csv + Shares.csv +
  InstantReposts.csv + Saved_Items.csv + Votes.csv + Rich_Media.csv +
  Events.csv produce timeline entries on the OPERATOR self-page only.
  NEVER creates new entity pages — activity rows referencing unknown
  entities get a `[[unresolved:<name>]]` placeholder + a review-queue
  entry. Reactions.csv is HARD-SKIPPED (operator decision S191-default-2).
- **Learning history** (Phase 6, T2): Learning.csv produces one stub
  per distinct course at `~/brain/media/courses/<slug>.md` + operator
  self-page timeline entries. Per-course frontmatter: `type: course`,
  `provider`, `topic`, `completed_on`, `duration_minutes`.
- **Preferences → USER.md** (Phase 7, T1): SearchQueries.csv +
  Ad_Targeting.csv + Inferences_about_you.csv + Ads Clicked.csv +
  LAN Ads Engagement.csv route to `~/.hermes/USER.md` sub-sections
  per `browser-history-bootstrap` precedent. NEVER writes to
  `~/brain/`. T1-by-storage but operator-facing identity-doc, not
  brain corpus.
- **Hard-skip list** (Phase 8): messages.csv + guide_messages.csv +
  learning_coach_messages.csv + learning_role_play_messages.csv +
  LearningCoachMessages.csv + Reactions.csv + Logins.csv +
  Jobs/Job Seeker Preferences.csv + SavedJobAlerts.csv +
  Receipts_v2.csv are NEVER opened. No `csv.reader` / `pd.read_csv`
  on these files. Hard-coded in SKIP_LIST module-level constant.
- **Idempotent on re-run**: Per-CSV checksum-skip + per-row
  stable-hash skip. Re-running on unchanged source = no-op
  (counters[idempotent_skip] > 0; counters[written] = 0).
- **No source mutation**: Read-only against the LinkedIn export
  directory.
- **No external API calls**: `external_inference: none` +
  `external_apis: []`. Verified by import inventory (no `requests`,
  `urllib.request`, `httpx` usage outside `_shared/git_commit.py`'s
  subprocess `git push`).
- **Dedup engine canonical**: Phase 2 + Phase 3 + Phase 4 share the
  same `dedup.py` module extending the apple-contacts D49 alias
  mechanism. Canonical identity precedence:
  `linkedin_url > email > DIN-5007-2-normalized name`.
- **Iron-Law back-linking**: Every stub created in Phase 2 + Phase 4 +
  Phase 6 has at least one inbound link from the operator self-page
  Timeline + the institution page (where applicable).
- **Commit + push at end** (Phase 9): Mirrors
  apple-contacts-bootstrap Phase 5 + enrich-orchestrator D107 pattern.
  Skill's own `commit_and_push_brain()` call absorbs the
  126-file T1-leak backlog + the new writes in a single commit pass.
  Dry-run skips this entirely.
- **Routing-config dependency**: This skill DEPENDS on
  `~/.hermes/sources/known-academic-orgs.yaml` +
  `~/.hermes/sources/family-group-terms.yaml` existing + parsing
  cleanly. Missing → fail-fast with explicit error message. Verified
  present at skill-author time (2026-05-23).

## Phases

### Phase 0: Routing-config preflight + source discovery

0. Verify `~/.hermes/sources/known-academic-orgs.yaml` AND
   `~/.hermes/sources/family-group-terms.yaml` exist + parse
   cleanly. Missing OR malformed → fail-fast.
1. Verify source directory exists at
   `~/resources/local-agent-system/data/LinkedIn/Complete_LinkedInDataExport_05-14-2026.zip/`
   (override via `--source <path>` CLI flag).
2. Enumerate present CSVs; cross-check against expected file list
   from the per-CSV routing contract. Log per-CSV `present` /
   `missing` / `skipped` status.
3. Resolve operator self-page: read `~/.hermes/USER.md` for
   `operator.brain_slug:` field if present; fall back to
   `emanuel.md`. If the resolved file doesn't exist in
   `~/brain/people/`, fail-fast — Phase 1 augments, never creates,
   the operator self-page.

### Phase 1: Operator identity ingest (operator self-page + institutions)

For each operator-identity CSV (Profile.csv, Positions.csv, etc.):

1. Compute per-CSV stable checksum (sha256 of file bytes). Skip if
   matches stored value in operator self-page frontmatter field
   `linkedin_bootstrap_checksums.<filename>`.
2. Parse rows via `parsers/operator_identity.py` →
   `OperatorIdentity` dataclass.
3. Update operator self-page:
   - Frontmatter: add `linkedin_member_url`, `linkedin_headline`,
     `linkedin_location` (compiled-truth fields). Add
     `linkedin_bootstrap_checksums:` map.
   - Compiled-Truth section: positions list + education list +
     skills list + languages list (auto-generated; user-authored
     content above the line preserved).
   - Timeline: per-position entry (`<start_date>–<end_date> · Role
     at [[institutions/<org-slug>]]`); per-education entry.
4. For each distinct past employer in Positions.csv: create or
   update `~/brain/institutions/<org-slug>.md` stub. Source label
   = `linkedin-bootstrap`.
5. For each distinct school in Education.csv: same as employer.

### Phase 2: Connections.csv → people/ stubs (with D49 alias dedup)

1. Compute per-CSV checksum; skip if unchanged.
2. Parse Connections.csv via `parsers/connections.py` — handles the
   LinkedIn CSV preamble strip (scan for header row starting with
   `First Name,` or `Email Address,`).
3. For each Connection row:
   - **Skip** LinkedIn-Member placeholder rows (first_name = "LinkedIn"
     AND last_name = "Member").
   - **Skip** rows with no name AND no URL AND no email →
     review-queue with `review_reason: linkedin-connection-no-identity`.
   - **Dedup against existing brain pages** per canonical
     precedence: `linkedin_url > email > normalized-name`.
   - **Match** → load existing page; add LinkedIn URL to `aliases:`
     if missing; add `connected_on` timeline entry; preserve all
     other content. Counter: `phase2_alias_merged`.
   - **No match** → resolve sub-dir per routing configs (academic
     org → `colleagues/`; otherwise → `personal/`); create stub
     with Connection data; backlink from operator self-page Timeline.
     Counter: `phase2_created`.
4. Idempotent re-run: stub already exists for this connection →
   no-op (counter: `phase2_idempotent_skip`).

### Phase 3: ImportedContacts.csv → alias-merge ONLY

1. Compute per-CSV checksum; skip if unchanged.
2. Parse ImportedContacts.csv via `parsers/imported_contacts.py`.
3. For each row:
   - **Dedup against existing brain pages** per precedence:
     `email > phone (E.164 normalized) > normalized-name`.
   - **Match** → add LinkedIn-side aliases (email + phone variants
     not already on page) to `aliases:`. Counter:
     `phase3_alias_merged`.
   - **No match** → log + skip. NEVER creates a new page.
     Counter: `phase3_no_match_skipped`.

### Phase 4: Network signals → relationship facts + timeline

For each network-signals CSV (Endorsements given/received, Invitations,
Company Follows, Member Follows):

1. Compute per-CSV checksum; skip if unchanged.
2. Parse via `parsers/network_signals.py` → typed `NetworkSignal`
   records.
3. Per signal: dedup the referenced entity per Phase 2 precedence;
   create stub if not found; add a fact + timeline entry capturing
   the signal type + date.

### Phase 5: Activity timeline → operator self-page ONLY

For each activity CSV (Comments_*.csv, Shares.csv, InstantReposts.csv,
Saved_Items.csv, Votes.csv, Rich_Media.csv, Events.csv):

1. Compute per-CSV checksum; skip if unchanged.
2. Parse via `parsers/activity.py`.
3. **HARD INVARIANT**: NEVER creates entity pages from activity rows.
4. For each activity row: write a single timeline entry on the
   operator self-page. If the row references an entity (e.g.,
   Comment on someone's post), and that entity is already in
   `~/brain/people/` or `~/brain/institutions/`, cross-link
   normally. If not, use `[[unresolved:<name>]]` + emit a
   review-queue entry.

**Reactions.csv is HARD-SKIPPED.** Reactions = low-signal high-volume
"liked X" rows; the operator scope decision (S191-default-2) is to
skip these entirely. The parser does not read this file even if
present.

### Phase 6: Learning.csv → media/courses/ stubs

1. Compute per-CSV checksum; skip if unchanged.
2. Parse via `parsers/learning.py` → `Course` records.
3. For each distinct course (dedup by course name + provider):
   create `~/brain/media/courses/<slug>.md` stub. Add operator
   self-page timeline entry: `<completed_on> · Completed
   [[media/courses/<slug>]] via <provider>`.

### Phase 7: Preferences → ~/.hermes/USER.md

Per `browser-history-bootstrap` v0.1.0 precedent. Reads
SearchQueries.csv + Ad_Targeting.csv + Inferences_about_you.csv +
Ads Clicked.csv + LAN Ads Engagement.csv. Aggregates by domain /
category / topic; appends sub-sections under canonical USER.md
top-level headings:

- **Topical interests** — top-N search-query topics + LinkedIn-side
  Inferences_about_you signal categories.
- **Vendors / SaaS** — Ad_Targeting + Ads Clicked → vendor categories.
- **Communities** — Company Follows + Member_Follows → professional
  communities.

NEVER writes to `~/brain/`. NEVER creates entity pages from this
phase. T1 storage (USER.md is T1 per its frontmatter) but
operator-facing identity-doc, not brain corpus.

## Anti-Patterns

- **Reading messages.csv body** — operator scope decision S191-Q1
  hard-skips messages entirely. No `csv.reader(open("messages.csv"))`
  anywhere. The SKIP_LIST is enforced by a unit test that scans the
  codebase for forbidden file references.
- **Creating new pages from ImportedContacts.csv** — operator scope
  decision S191-Q2: alias-merge ONLY against existing pages. The 318
  KB CSV is mostly stale phone-upload backup; creating thousands of
  ghost pages would pollute the corpus.
- **Creating new pages from Activity rows** (Phase 5) — Comments /
  Shares / Reactions reference posts on the LinkedIn network; the
  authors of those posts may not be your connections. Creating a
  stub for every commented-on author would pollute `people/`.
  Phase 5 timeline-only.
- **Overwriting user-authored body content during operator self-page
  update** — `~/brain/people/emanuel.md` may have hand-written
  content above the line. Re-runs MUST preserve everything except
  the auto-generated `## LinkedIn Identity` sub-section + Timeline
  entries.
- **Slugifying URLs as identifiers** — LinkedIn member URLs contain
  trailing numeric IDs (`/in/klaus-mueller-12345`) that change when
  LinkedIn re-canonicalizes. The dedup engine treats the URL as an
  alias signal, NOT as the slug-key. Slug-keys are still
  D-DD2-compliant DIN 5007-2 kebab-case of the display name.
- **Auto-translating CSV column names** — LinkedIn ships some CSVs
  with multi-line preamble (a `Notes:` section before the real
  header row). Per-CSV parsers detect the real header by scanning
  for the canonical first-column name (e.g., `First Name,` for
  Connections.csv) rather than assuming row 0 is the header.
- **Calling external APIs** — `external_inference: none` +
  `external_apis: []`. No LinkedIn API scrape, no enrichment-side
  HTTP calls. Future v0.2.0+ may add local-LM-Studio disambiguation;
  changes must go through the full
  `_conventions-tiers-and-egress.md §5` opt-in protocol.

## Test Corpus Shape

**Per CDD §2.1 golden data:**
- 1 row per cluster type (Profile + Positions + Education + Skills +
  Languages + Email + Phone + Registration + Verifications + Profile
  Summary).
- 1 known dedup-target person (existing brain page with matching
  apple-contacts-style alias; LinkedIn URL added on merge).
- 1 known dedup-target org (existing institutions/ stub).
- 1 unknown person → creates new `people/colleagues/<slug>.md` stub.
- 1 unknown org → creates new `institutions/<slug>.md` stub.
- 1 LinkedIn-Member placeholder row → skipped.
- 1 activity row referencing unknown entity → routes to review-queue,
  no new entity page.

**Edge cases:**
- Connections.csv multi-line preamble (LinkedIn's
  `"Notes:\n\nWhen exporting your connection data, ..."` block before
  the real header).
- Row with no email AND no URL AND no name → review-queue.
- Connection with same email as existing brain page → alias-merge,
  not new page.
- Connection with same DIN-5007-2-normalized name as existing
  brain page but different email → name-only match, route to
  review-queue per `cross-source-name-only-match` precedent.

**Phase-skip cases:**
- messages.csv present in source dir → never opened by Phase 8.
  Verified by codebase grep + unit test.
- Reactions.csv present in source dir → never opened.
- Re-run on unchanged source → 0 writes, all phases idempotent_skip.
