## Architectural framing

**LinkedIn export = canonical primary source for the operator's professional-network subgraph.** Every person + institution in the brain that has any LinkedIn-side signal gets richer texture from this same export — not just the operator self-page.

The retired `enrich-orchestrator` was conceived as external-research enrichment (web search + LLM synthesis for thin stubs). linkedin-deepen is **the opposite**: pure exploitation of already-on-disk LinkedIn export data into existing entity pages.

## Contract

This skill guarantees:

- **Operator identity deepen** (Phase 1, T2): the operator self-page at `~/brain/people/emanuel.md` (or `<brain-root>/people/<operator-slug>.md`) gains skill-owned sections — `## Executive Summary`, `## Career Arc (from LinkedIn)`, `## Education (from LinkedIn)`, `## Skills (from LinkedIn)`, `## Languages` — sourced from Profile.csv + Positions.csv + Education.csv + Skills.csv + Languages.csv. User-authored body content above the line preserved byte-for-byte.

- **Per-connection deepen** (Phase 2, T2): each `people/<X>.md` page with a `linkedin_url` alias (set by linkedin-bootstrap Phase 2 or pre-existing colleague enriched via alias-merge) gains a `## LinkedIn Profile Snapshot` section with verbatim Position + Company + Connected On from Connections.csv. State paragraph composed from those fields (deterministic template OR LLM-prose). NO new pages created.

- **Endorsement edges** (Phase 3, T2): Endorsement_Given_Info.csv + Endorsement_Received_Info.csv rows → bidirectional timeline entries on both endpoints (counterparty page AND operator self-page). Cross-link via `Endorsee/Endorser Public Url`. Counterparty missing → log + skip; do NOT create. gitleaks-safe format: `Endorsement given/received: <Skill> (via LinkedIn)`.

- **Invitation edges** (Phase 4a, T2): Invitations.csv rows with `inviterProfileUrl` or `inviteeProfileUrl` → direction-aware bidirectional timeline entries with Message text if present.

- **Follow signals** (Phase 4b, T2): Member_Follows.csv (name-only) → name-matched `people/<X>.md` get `followed_by_operator: true` frontmatter; Company Follows.csv → name-matched `institutions/<X>.md` get same. Non-matches logged + skipped (per operator S191-D119+1 scope lock: match name-only against existing pages; no new pages).

- **Operator voice** (Phase 5, T2): Comments_*.csv + Shares.csv → `emanuel.md` Timeline + `## What They Engage On (LinkedIn signal)` + `## What They Amplify (LinkedIn shares)` sections. NO counterparty cross-link attempted (operator scope lock — no regex-name-match against Message body; avoids false-positive attributions).

- **Institution deepen** (Phase 6, T2): Positions.csv Description + Education.csv Notes/Activities → matching `institutions/<X>.md` "Operator's Role Here" section. Company Follows → `followed_by_operator: true` frontmatter. Connections.csv aggregated per Company → `## LinkedIn Connections Employed Here` graph-signal section with count + top-N list.

- **Idempotent**: per-(page-path, section-header, source-row-hash) stable-hash registry at `audit/applied.json`. Re-run on unchanged source = 0 mutations.

- **NO new pages created. EVER.** Structurally bounded — `pages_created` counter asserted == 0 in test suite.

- **NO body-content overwrite.** Sections written by this skill (the named `## <skill-owned>` headers above) are the ONLY sections subject to replace-on-rerun. Everything else is preserved.

- **Iron-Law backlinks.** Endorsement + invitation edges are bidirectional (both endpoints get matching timeline entries with same date + counterparty reference).

- **Privacy-tier inheritance.** Pages retain their existing `privacy_tier`; never escalate or downgrade.

- **gitleaks-safe phrasing.** Reuses linkedin-bootstrap S191-D119+1 post-patch format: `(via LinkedIn)` suffix on timeline lines; no `linkedin_*: "<hex>"` frontmatter patterns; static-codebase-check test enforces.

- **SKIP_LIST hard invariant.** messages.csv + Reactions.csv + 8 other CSVs are NEVER opened by any phase. Sibling invariant to linkedin-bootstrap.

- **Commit + push at end.** Same `commit_and_push_brain()` pattern as linkedin-bootstrap Phase 9 (D107 / S191-D119 footprint). Dry-run skips all writes including commit.

## Phases

### Phase 0: Preflight

Verify source dir exists (extracted LinkedIn export) + operator self-page resolvable. Soft-warn if linkedin-bootstrap hasn't run (no LinkedIn URL aliases found in any people page) but proceed (operator can still deepen the self-page from CSVs).

### Phase 1: Operator self-page deepen

Read Profile + Positions + Education + Skills + Languages. Compose + write skill-owned sections into `<operator-page>`:

- `## Executive Summary` — Profile.csv Summary verbatim (or LLM-rewritten if `--llm-prose-local`).
- `## State` — augmented with current Position + Company + Location + Industry (append to existing State; preserve).
- `## Career Arc (from LinkedIn)` — chronological Positions; per-position narrative if Description present; structural-only if empty.
- `## Education (from LinkedIn)` — Education entries with degree + notes + activities.
- `## Skills (from LinkedIn)` — Skills list deduped against existing brain tags.
- `## Languages` — Languages list.

### Phase 2: Per-connection deepen

For each Connections.csv row: resolve target page via `find_existing_person_by_linkedin_url` (from linkedin-bootstrap). If found, write `## LinkedIn Profile Snapshot` section (Position + Company + Connected On + URL) and compose a State paragraph (deterministic OR LLM). LinkedIn-Member placeholder rows skipped silently. No new pages.

### Phase 3: Endorsement edges (bidirectional)

For each Endorsement_Given_Info row + each Endorsement_Received_Info row: resolve counterparty via `Endorsee/Endorser Public Url`; emit bidirectional timeline entry on counterparty + operator self-page using format `Endorsement given/received: <Skill> (via LinkedIn)`. Counterparty page missing → log + skip.

### Phase 4: Invitations + Follows

- **4a Invitations**: per row with profile URL, resolve counterparty; emit direction-aware bidirectional timeline entry; include Message text if present (truncated to 200 chars).
- **4b Member_Follows**: name-only match against `people/**/*.md`; set `followed_by_operator: true` frontmatter on matches; write to operator self-page Watch List section.
- **4b Company_Follows**: name-only match against `institutions/**/*.md`; same frontmatter + Watch List.

### Phase 5: Operator voice

Comments_*.csv + Shares.csv → `<operator-page>` Timeline + `## What They Engage On (LinkedIn signal)` + `## What They Amplify (LinkedIn shares)` sections. NO entity-page writes from these CSVs. NO counterparty cross-link.

### Phase 6: Institution deepen

For each institution touched by Positions/Education/Company Follows:

- `## Operator's Role Here` — Position Description (or Education Notes/Activities for schools).
- `## LinkedIn Connections Employed Here` — count from Connections.csv aggregation; ranked list of top-20.
- `followed_by_operator: true` frontmatter if matches Company Follows.

### Phase 7: SKIP invariant verification

Verify SKIP_LIST files are present (informational) but never opened. Sibling to linkedin-bootstrap Phase 8.

### Phase 8: Commit + push

Real run only (skipped on dry-run + on non-default brain-root). Uses `_shared/git_commit.commit_and_push_brain` per D107 pattern.

## Anti-patterns

- ❌ DO NOT create new pages in any phase. linkedin-bootstrap is the only skill in this family that creates pages. linkedin-deepen ONLY enriches existing.
- ❌ DO NOT call external APIs. No `requests`, no `anthropic`, no LinkedIn API. Local-only.
- ❌ DO NOT open SKIP_LIST files. Static test scans codebase for any reference to `messages.csv` outside SKIP_LIST constants.
- ❌ DO NOT overwrite user-authored body content. Skill-owned sections (named in the Contract above) are the ONLY rewrite-on-rerun targets.
- ❌ DO NOT regress to `linkedin_*: <hex>` frontmatter patterns. Use `profile_csum:` style without prefix.
- ❌ DO NOT bypass idempotency. Every section write checks the applied.json registry first.
- ❌ DO NOT use `--llm-prose` (Anthropic). Use `--llm-prose-local` (qwen3.6-35b via LM Studio) instead.
